# docs/318 — the keep-gate ablation: measure what the verdicts buy (the ratchet curve)

> **Status:** P1–P2 executed 2026-06-12 — the adjudicator is
> `dos verify docs/318_keep-gate-ablation-plan P2`, never this sentence.
> P3–P4 are designed, not started. Tracking handle: issue
> [#21](https://github.com/anthony-chaudhary/dos-kernel/issues/21).

## The problem — the kernel has not met its own bar

Every DOS verdict is justified by "don't believe the self-report." But the
claim *"a loop that honors the verdicts out-produces a loop that ignores
them"* has itself never been measured. The kernel's own `improve()` doctrine
is KEEP only on a strictly env-measured gain; applied to the kernel as a
whole, that bar is unmet — no A/B exists comparing a gate-honoring loop
against a self-certifying control. What the verdicts buy is argued, not
witnessed. That is issue #21, and its loose done-condition is exact:

> a numbered design plan exists for the ablation, and its first phase
> produces a committed evidence file with measured deltas between the two
> arms — after which "what the verdicts buy" is a number, not a feeling.

## The shape — one boolean apart, refereed

The issue's owner comments fix the framing: run the **same** self-improving
recipe loop twice, identical in every way except the keep rule, and plot the
env-measured metric per cycle — **the ratchet curve**. The thesis predicts a
monotone staircase (gate honored) against a mean-reverting walk (loop
believes itself) whose *narration* still trends up.

The arena is the one the second owner comment names: **an AI training
loop** — "nanoGPT-speedrun-at-nano-scale, refereed" — because there the
metric is *genuinely noisy* (seed variance occurs naturally, no contrivance),
the eval harness is part of the editable tree (the #35 tamper surface is
load-bearing), and the bait taxonomy is pre-registered from the field
(cached outputs, persistent state, harness exploitation), not invented.

P1 keeps the arena but shrinks it to the laptop-cheap core: a
**character-level n-gram language model** trained on a seeded synthetic
corpus, where one full train+eval is milliseconds, so a 50-cycle, 3-arm
ablation runs in seconds and is reproducible byte-for-byte from one master
seed. This is a real training loop — parameters fit on a train split, quality
read as held-out negative log-likelihood (bits/char) — with the real failure
gradient the self-certifying arm needs: **capacity mutations (order up,
smoothing down) always look good in-sample and eventually hurt held-out**.
The self-grading loop is not a strawman that keeps everything; it is a loop
that grades its own homework on data it can see, which is exactly the failure
`improve()` exists to refuse.

## The design (P1)

**Task.** A pseudo-English corpus (~120k chars) generated by a seeded
word-vocabulary source — learnable character structure up to roughly the
word length, noise beyond it, so an over-capacity model has something real
to overfit. Per evaluation seed, disjoint train/val windows are sampled from
the corpus; seed variance in the measured NLL is the natural metric noise.

**Recipe.** `(order, add_k)` — the n-gram context length and the add-k
smoothing constant. The proposer mutates one parameter per cycle (order ±1,
add_k scaled), seeded identically across arms.

**The two channels, kept apart (the docs/138 split):**

- *Forgeable channel* — the proposer's own estimate: it trains the candidate
  on its own window and compares **in-sample** NLL against the incumbent's.
  This is the agent-authored claim; it rides `narrated` and is systematically
  optimistic about capacity.
- *Witness channel* — the referee's measure: **held-out** NLL averaged over K
  fresh seeded windows, mapped to integer work points against the cycle-0
  recipe's reference NLL. Bytes the proposer does not author.

**Arms (one decision rule apart, same proposer, same mutation stream):**

| Arm | Keep rule |
|---|---|
| **A — gated** | `dos.improve.classify` over the referee's K-seed work vs the carried baseline; KEEP/REVERT/ESCALATE honored as shipped (ESCALATE stops the arm — the RSI human-judgment seed) |
| **B — self-certified** | keeps iff its own in-sample estimate claims improvement; the referee still measures everything *for the record*, but the decision never reads it |
| **C — gated, single-sample** | arm A's rule with K=1 — the shipped gate exactly as issue #34 worries about it: noise can mint a KEEP and poison the carried baseline |

Arm C is the in-design ablation of #34 itself: A vs C isolates what the
multi-sample measurement discipline buys; A vs B isolates what the gate buys.
No kernel change is needed or made — the noise control lives in the
*measurement* the driver feeds the gate, which is precisely where docs/280
put the I/O boundary.

**Outputs (the committed evidence file):** `benchmark/improve_ablation/RESULTS.md`
(+ `results.json`) carrying, per arm: the ratchet curve (true held-out
quality of the incumbent per cycle, referee-measured on fixed plot seeds),
the cumulative *claimed* gain curve, kept/reverted/escalated counts,
**over-claims kept** (kept cycles whose claim the referee refutes), and the
final incumbent re-measured on fresh seeds (the real gain per arm). The
headline deltas the issue asks for are the final-gain gap A−B and the
over-claims-kept counts.

**Determinism.** All randomness flows from one `--seed`; the committed
results are reproducible by the one command in RESULTS.md. No wall-clock, no
network, stdlib + `dos` only. The bench imports `dos` (the allowed
direction); nothing under `src/dos/` knows it exists.

## Phases

### P1 — the refereed recipe loop + the committed evidence file

`benchmark/improve_ablation/`: the task (`task.py`), the three-arm runner
(`run_ablation.py`), its tests (`test_improve_ablation.py`, run by path —
`benchmark/` is outside the suite's `testpaths`), and the committed
`RESULTS.md` + `results.json` from the default-seed run.

**Done when:** the evidence file is committed with measured deltas between
the arms, `python -m pytest benchmark/improve_ablation -q` is green, and
re-running `python -m benchmark.improve_ablation.run_ablation --seed 0`
reproduces `results.json`. This phase is the issue's loose done-condition.

### P2 — the bait taxonomy (the #35 surface)

Seed the workspace with reachable instances of the field-observed hack
classes (a readable val split, a cache that persists across candidates, an
editable eval script) and a proposer that occasionally takes them. Add the
**bait scoreboard** (hacks kept vs refused, per class, per arm) to the
evidence file. This is the validation bed for the #35 witness-tamper floor:
today's gate KEEPs a tamper that fakes the metric — the scoreboard makes
that hole countable before and after #35 ships.

### P3 — over-claim drift, adjudicated by the shipped auditor

Make each cycle a commit in a throwaway git repo (subject = the proposer's
claim), and run `dos commit-audit --sweep` over each arm's history: the
drift rate per arm, measured by the same out-of-loop witness this repo uses
on itself. The "narration trends up while the metric doesn't" half of the
owner's chart becomes a number the existing tooling computed.

### P4 — scale the arena

Real text (tinyshakespeare-scale), an LLM proposer with billed tokens (the
`tokens per witnessed point of gain` column joins via the efficiency rung),
and the fleet-scale framing of the original issue body (commit-audit drift
rate and reconcile QUIET_INCOMPLETE rate as fleet metrics). Registration in
`benchmark/registry.py` joins here if the bench grows arms/prereqs worth
standardizing.

## Litmus

- **Kernel untouched.** The ablation consumes `dos.improve` as shipped; a
  finding about the gate (the #34 wedge, the #35 hole) files back into those
  design issues, never as a bench-side patch to the kernel.
- **The keep-bit reads zero loop-authored bytes in arm A.** Pinned by a test:
  every arm-A keep has referee work strictly above the carried baseline.
- **The forgeable channel is carried, never parsed** — except in arm B,
  where parsing it *is the ablation*.
