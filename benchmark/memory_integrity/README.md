# memory_integrity — do the memory gates catch bad memories, and at what cost?

**Research question** — given a stream of candidate memories about a real
repo, how much poison does the write gate (`dos memory admit`, docs/314)
refuse, how much decay does the recall gate (`dos memory recall`, docs/103)
catch, and what does that distrust cost in falsely-refused truth?

Design and taxonomy: [docs/316](../../docs/316_bad-memory-taxonomy-and-integrity-benchmark-plan.md).
Numbers: [RESULTS.md](RESULTS.md) (regenerated by every run).

## Run it (free — git only, no model, no network)

```bash
python -m benchmark.memory_integrity.run            # writes RESULTS.md + results.json here
python -m benchmark.memory_integrity.run --json     # full JSON to stdout
```

Or through the standardized runner: `python -m benchmark._run run memory_integrity`.

## How it stays honest

- **Labels are env-authored.** `corpus.py` BUILDS the scratch git repo, so each
  candidate's truth value is fixed by constructed commits — never by a rater.
- **Two time points.** Candidates are admitted at T0, the repo then mutates,
  stored memories are recalled at T1 — so the aged-stale class must PASS the
  write gate (it was true) and be CAUGHT by the recall gate (it isn't anymore).
- **The misses are headline rows.** Grammar-evasive poison (prose-shaped lies)
  is EXPECTED to pass admission today; that row is the published ceiling and
  the docs/316 P2 work-list, not a buried caveat.
- **The baseline is the industry default.** `admit-all` is what every
  auto-extracting memory pipeline does: store the narration. Poison admission
  is 100% by construction — the comparison, not a strawman.

`tests/test_memory_integrity_bench.py` pins one candidate per class so the
kernel suite catches a drifting gate or corpus.
